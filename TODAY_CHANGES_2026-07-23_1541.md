## 1. Demo 稳定性修复

### 1.1 关闭 API 的热重载，减少 SSE 断线

修改文件：

- `apps/api/src/pinch_api/app.py`

修改内容：

- 将 demo 启动时的 `uvicorn.run(..., reload=True)` 改为 `reload=False`

原因：

- 之前 API 开了热重载
- 文件一变动就会重启服务
- 前端通过 EventSource 订阅 `/api/runs/<id>/events`
- API 一重启，SSE 就会断开，页面容易卡在：
  - `Preparing the next Router decision`
  - `Model provider is working on the next Agent step`
  - `Scoring the completed run...`

目标：

- 先让 demo 演示稳定
- 牺牲热更新，换取不轻易断线

---

### 1.2 前端 SSE 出错时不再直接 `close()`

修改文件：

- `apps/web/src/main.tsx`

修改内容：

- `source.onerror` 不再立即 `source.close()`
- 改成出错后回查：
  - `/api/runs/<run_id>`
  - `/api/runs/<run_id>/events/history`

效果：

- 即使 SSE 断了一次
- 前端也会尝试重新同步当前 run 状态和历史事件
- 如果 run 已结束，会补齐 completed / failed 状态

## 2. Windows 编码问题修复

### 2.1 修复 run 历史 / 元数据读写时的编码问题

修改文件：

- `apps/api/src/pinch_api/service.py`

修改内容：

- 所有 `read_text()` / `write_text()` 改为显式使用 `encoding="utf-8"`

主要影响：

- `run.json`
- `events.jsonl`
- run history / summary 统计

原因：

- Windows 默认编码可能走 `gbk`
- 一旦模型输出里有：
  - `—`
  - `’`
  - 其他非 ASCII 字符
- 就可能导致：
  - run history 读取失败
  - grading 阶段异常
  - 最终状态无法写回 completed

---

### 2.2 修复 PinchBench 自动评分时读取 workspace 文件的 `gbk` 问题

修改文件：

- `pinchbench-skill/scripts/lib_grading.py`

修改内容：

- 在执行 task 内嵌 automated grader 前
- 将常见的：
  - `.read_text()`
- 自动改写成：
  - `.read_text(encoding='utf-8', errors='replace')`

原因：

- 很多 task 的 grader 代码里直接 `Path.read_text()`
- Windows 默认 `gbk` 读取模型生成的 UTF-8 文件时容易炸
- 典型表现：
  - 卡在 `grading_started`
  - 页面一直显示 `Scoring the completed run...`

效果：

- 评分阶段在 Windows 下更稳定
- 不容易因为 Unicode 字符导致 run 卡住不收尾

## 3. Workspace 初始化兼容性修复

修改文件：

- `apps/agent/src/pinch_agent/cases.py`

修改内容：

- `prepare_workspace()` 以前只支持：
  - `{"source": "...", "dest": "..."}`
- 今天改成兼容多种格式：
  - 纯字符串路径
  - `source/dest`
  - `path`
  - `content` 内联文件
  - 目录拷贝

原因：

- 某些 case 的 `workspace_files` 元数据格式不统一
- 之前会报：
  - `KeyError: 'source'`
- 结果就是：
  - 你点击 `Start agent run`
  - 后端在 prepare_workspace 阶段直接 500
  - 页面看起来像“点了没反应”

效果：

- 新建 run 时对 task 元数据更宽容
- 减少 case 因 workspace 初始化格式差异而直接失败

## 7. D 盘缓存 / 临时目录迁移

今天已经做的环境配置：

- 用户级 `UV_CACHE_DIR` -> `D:\tmp\uv-cache-yesq6`
- 用户级 `TEMP` -> `D:\tmp\temp-yesq6`
- 用户级 `TMP` -> `D:\tmp\temp-yesq6`

说明：

- 这是低风险做法
- 没有动系统级分页文件设置
- 后续新的 `uv` 缓存和大部分临时文件会优先走 D 盘

注意：

- 已经打开的旧进程可能仍在使用旧的 C 盘路径
- 新开终端 / 新启动的进程才会稳定使用新的用户环境变量
