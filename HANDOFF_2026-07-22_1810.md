## 这次主要改了什么

### 1. Demo 已接入本地 Router

当前 demo 不是固定模型调用，而是会先经过本地 Router 做选择，再调用真实模型。

Router 运行依赖以下本地资产:

- `apps/router/var/assets/chinese_mobilebert_base_f2`
- `apps/router/var/assets/best_router_finetuned.pth`
- `apps/router/var/assets/runtime_model_representations.npz`

说明:

- 现有 `.pth` checkpoint 是按 **7 个 canonical labels** 训练的
- 运行时默认候选已限制为 **2 个模型**
  - `deepseek-v4-flash`
  - `deepseek-v4-pro`
- 为了兼容 checkpoint 维度，配置中仍保留 7 维 label 定义，但 API 默认只传这 2 个候选给 Router

### 2. 模型配置改成了 DeepSeek 两模型

目标模型改为:

- `DeepSeek-V4-Flash`
- `Deepseek-V4-Pro`

注意:

- `Pro` 的可用模型名大小写经过验证，当前应使用:
  - `Deepseek-V4-Pro`
- 之前的 `DeepSeek-V4-Pro` 会报:
  - `error model name`

### 3. Demo 前端显示做了美化

改动点:

- `Task completed` 不再直接显示一大坨工具 JSON
- `write_file/read_file/list_files` 结果会以更结构化的方式展示
- 大段 `write_file` 内容默认折叠，不再直接把整段文本铺满页面
- waiting 文案从固定 `MiniMax` 改为更通用的 model provider 文案

### 4. 修复了失败 run 的 summary 统计问题

之前失败 run 容易显示:

- `cost = 0`
- `No tools`
- `routed models = —`

现在已经修正为:

- 失败 run 也会写入 summary
- summary 按 `router_decision` 统计步骤和 cost
- 工具调用会正确统计进 `tools`

### 5. 修复了二进制文件被 read_file 读成乱码的问题

之前像 `.xlsx/.docx/.pptx/.zip/.pdf/.png` 这类文件会被当成文本硬读，页面出现乱码。

现在 `read_file` 会识别常见二进制文件，并返回友好提示，例如:

```text
binary file: company_expenses.xlsx (preview skipped)
```

## 已知现象 / 注意事项

### 1. 最后一步有时会看起来卡很久

这通常不是 Router 卡住，而是最后一轮真实模型调用在吃完整历史。

原因：

- Router 看的是压缩后的 `RoutingContext.text`
- 但 provider 调用吃的是完整 `trace.messages`
- 如果之前 `read_file` 或 `write_file` 带了很多内容，最后一轮会很重

### 2. 目前还没有做“write_file 后历史瘦身”

目前系统仍可能把较大的 assistant 内容继续保留在后续模型历史里。

更推荐的后续优化方向：

- `write_file` 后续历史只保留:
  - `wrote xxx`
- 不再把整份文件内容继续塞回下一轮模型上下文

### 3. 7 维 checkpoint 与 2 模型运行候选并存

这不是 bug，而是当前实现上的兼容方案：

- checkpoint 训练维度仍是 7
- 运行时只传 2 个候选给 Router

## 建议后续优化项

以下是当前比较明确、适合下一位维护者继续优化的方向。

### 1. Case 详情里的 `AUTOMATED · 90S` 这类信息展示不够直观

当前 case 详情区域会直接展示诸如：

- `AUTOMATED`
- `90S`

这类信息，但对不熟悉 PinchBench 的用户来说不够直观，容易“不明所以”。

建议：

- 若信息价值不高，可直接隐藏
- 或改成更容易理解的文案，例如：
  - `评分方式：自动`
  - `超时时间：90 秒`
- 或做成 tooltip / 说明标签，而不是生硬拼接在标题下

### 2. 最好支持按 case 类别筛选

目前 case 本身是有 `category` 的，例如：

- `analysis`
- `coding`
- `research`
- `productivity`

但前端当前主要还是靠搜索。

建议：

- 在 case 列表上方增加类别筛选标签 / tab / dropdown
- 支持：
  - 查看全部
  - 单类别筛选
  - 最好能和关键词搜索叠加

这样更方便演示或只看某一类任务表现。

### 3. 最好做“两路对比”

当前 demo 只有一路：

- `经过 Router 选择候选模型，再执行对应模型`

建议增加另一路基线：

- `不经过 Router，直接固定调用最贵模型 Deepseek-V4-Pro`

理想效果：

- 同一个 case 能看到两路结果并排对比
  - Router 路线
  - 固定 Pro 路线
- 比较维度至少包括：
  - 总 cost
  - 总耗时
  - 步数
  - 是否成功
  - 最终输出差异（可选）

这样才能更直观展示：

- Router 是否真的节约了 cost
- 在保证效果的前提下是否减少了高价模型使用

如果后续继续做评估页，建议直接显示：

- `Router total cost`
- `Always-Pro total cost`
- `Cost saved`
- `Saved percentage`

### 4. 有些 case 在最后一步会卡很久

这是当前体验里比较明显的问题。

已知原因大概率不是 Router 慢，而是：

- 最后一轮真实模型调用会吃完整历史
- 历史里可能包含：
  - `read_file` 返回的大段文本
  - `write_file` 时 assistant 输出的大块内容
  - 工具结果消息

建议优化方向：

- 工具结果做摘要化，不要把大段内容原样喂回下一轮模型
- `write_file` 后续历史尽量只保留：
  - `wrote xxx`
- 对 `read_file` 的大文本结果做截断或摘要
- 前端 waiting 态增加更明确的反馈，例如：
  - 已等待 xx 秒
  - 当前正在等待模型返回
  - 当前上下文较大，响应可能较慢

如果只做一个优先级最高的优化，建议先做：

- **减少后续模型调用历史中的冗余大文本**

这会同时改善：

- 最后一步卡顿
- token 成本
- 页面等待体验

### 5. 可选的补充优化

除以上 4 点外，还可以考虑：

- 对失败 run 的展示更明确区分：
  - Router 选择问题
  - Provider 调用失败
  - 工具执行失败
- 给 run summary 增加：
  - 实际执行模型统计
  - 每步 latency
  - 更明显的成功 / 失败原因
- 对二进制文件增加前端层面的友好提示样式，而不是只显示纯文本：
  - `binary file preview skipped`
  - `Excel workbook detected`
